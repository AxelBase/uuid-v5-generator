What is a UUID v5 and Why Use It?

Published: November 28, 2025

A UUID v5 is a 128-bit identifier generated by applying SHA-1 to a namespace UUID and a name. Unlike random UUIDs (v4), v5 is fully deterministic: the same namespace + name combination will always produce the exact same UUID — forever, across any system, anywhere in the world.

This makes UUID v5 uniquely powerful in distributed architectures where consistency and reproducibility matter more than pure randomness.

How UUID v5 Works

The process is defined in RFC 4122:

  1. Take a namespace UUID (e.g., the official DNS namespace: 6ba7b810-9dad-11d1-80b4-00c04fd430c8)
  2. Combine it with any string (the "name") — like a domain, URL, username, or file path
  3. Run SHA-1 on the combined bytes
  4. Take the first 128 bits, set version = 5 and variant = RFC 4122
  5. Result: a perfectly valid, globally unique, deterministic UUID

Why Choose v5 Over v4 or v1?

While UUID v4 is excellent for public tokens and v1 offers sortability, v5 combines the best of both worlds when you need:

  • Reproducibility — migrate data, rebuild caches, sync systems with zero drift
  • Human-readable mapping — know exactly which input created which ID
  • No coordination — generate consistent IDs without a central authority
  • Security through obscurity not required — names can be public (e.g., URLs)

Real-World Use Cases

UUID v5 shines in:

  • Content-addressable storage (IPFS, Git)
  • Cache keys from URLs or usernames
  • Entity IDs in microservices ("user:alice@example.com" → same UUID everywhere)
  • Configuration identifiers across environments
  • Event sourcing and CQRS systems

FAQ

Is UUID v5 still secure?

Yes — when used correctly. The namespace acts as a salt. Never use sensitive data as the name.

Can two different names produce the same v5 UUID?

Only with an astronomically unlikely SHA-1 collision (theoretical, not practical).

Should I use v5 as a primary key?

Absolutely — especially in distributed databases. It prevents ID conflicts during merges.

UUID v5 isn't just another identifier — it's a contract: same input, same output, everywhere, forever.
